The case for domain-driven design systems

Most design systems fail because they model UI components instead of business domains. Here's what happens when you flip that assumption.

Jeremy Fuksa
Design leader & systems thinker

There's a pattern I keep seeing in design system work: a team builds a beautiful component library, documents it thoroughly, ships it to production — and then watches adoption plateau at 40%. The components are well-made. The documentation is clear. But something isn't clicking.

The component-first trap

The problem is almost always the same. The system was built around visual primitives — buttons, inputs, cards, modals — instead of around the domain concepts that the product team actually thinks in. A "card" means nothing to a product manager. An "account summary" means everything.

This isn't a new insight in software engineering. Domain-driven design has been a foundational practice in backend architecture for decades. But design systems are still stuck in the component-primitive era.

The map is not the territory, but a good map should use the territory's language.

What domain-driven looks like in practice

Instead of starting with a button audit, start with a domain model. What are the core objects in your product? What actions can users take on them? What states do they move through?

// Instead of this:
<Card variant="outlined" size="md">
  <CardHeader>...</CardHeader>
  <CardBody>...</CardBody>
</Card>

// Try this:
<AccountSummary
  account={account}
  showActions={permissions.canEdit}
/>

The second version embeds domain knowledge directly into the component API. It's opinionated, which is exactly what makes it useful.

A note on flexibility

The most common objection is that domain components are too rigid. This misunderstands the goal. You still have primitive components underneath — domain components are compositions that encode your team's best understanding of how concepts should be presented.

  • Primitives handle the how — spacing, color, typography
  • Domain components handle the what — accounts, transactions, workflows
  • The domain layer is where adoption actually happens

Measuring success differently

When you build domain-first, your success metric shifts from "component coverage" to "concept coverage." You're not asking "do teams use our button?" — you're asking "do teams model accounts the same way across every surface?"

The results speak for themselves. Teams that ship domain components alongside primitives see 2-3x higher adoption in the first year, because the components match the mental model product teams already have.

This is the approach behind Domain Foundation — the consulting methodology I'm building for organizations that want design systems that actually stick. More on that soon.
